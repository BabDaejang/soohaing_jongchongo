## [배치 1 프롬프트] — useSequentialRun 동시성 opt-in + 다단 연쇄 + 재시도 계약

`docs/리팩토링_3.md` 배치 1을 실행하라. 범위: 훅 확장(동시성 풀·다단 연쇄·retryable 재시도·건별 소요 시간 로그) + LLM 레이트리밋 에러의 status 보존. **UI·서버 액션 개편은 하지 않는다**(배치 2·3). `docs/리팩토링_3_프로그레스.md`를 이 문서의 관행(리팩토링_2_프로그레스.md 참조)대로 신설한다.

### 1단계 — 훅 확장 (`lib/hooks/use-sequential-run.ts`)

파일 전체를 먼저 읽고, 다음 의미론을 구현한다. **기존 소비자 무변경 동작이 최우선** — concurrency 미전달(=1)이면 현행과 로그 순서·pause/stop·서킷 브레이커·progress까지 관측 가능 동작이 동일해야 한다.

1. **동시성 풀**: `concurrency?: number` 인자(1~3 클램프, 기본 1). `start()` 시점에 읽어 한 실행 전체(모든 스테이지)에 고정. 풀 규칙 — 항상 최대 `concurrency`개의 stepOne Promise를 유지하고, 하나가 끝나면 다음 대기 건을 즉시 투입한다(Promise.allSettled 배치 단위가 아니라 슬라이딩 풀). 완료 순서대로 로그·progress를 갱신하고, 로그 라인은 기존처럼 대상 label을 접두해 교차 출력을 식별 가능하게 한다.
2. **일시정지(pause)**: 새 투입만 멈추고 진행 중 건은 완료를 기다린다(기존 ref 플래그 + 폴링 관행 유지). 재개 시 대기 건부터 이어간다.
3. **긴급중단(stop)**: 새 투입 중단 → 진행 중 건의 응답을 기다린 뒤(강제 취소 없음 — 서버 액션은 어차피 완주한다) 처리분까지 succeeded/failed에 반영하고 `aborted: true`로 finalize를 호출한다. **finalize는 반드시 모든 in-flight가 정산된 후 1회만.**
4. **서킷 브레이커**: `maxConsecutiveFailures`(기본 3)는 **완료 순서 기준의 연속 최종 실패**로 판정한다. 도달 시 새 투입을 멈추고 in-flight를 정산한 뒤 stop과 동일하게 종료한다. 성공 1건이 카운터를 리셋하는 기존 규칙 유지.
5. **재시도 계약**: `StepResult`에 `retryable?: boolean` 선택 필드 추가. `retryable: true`인 실패는 같은 대상에 대해 최대 2회 재시도(백오프 2초 → 6초, 재시도 중 stop 플래그를 확인해 즉시 탈출)하고, 재시도 소진 후의 실패만 failed·연속 실패 카운터에 집계한다. 재시도는 터미널에 "재시도 n/2 — {사유}" system 라인으로 표시.
6. **다단 연쇄 일반화**: `RunPlan`에 선택 필드 `nextStage?: (r: RunOutcome) => RunPlan | null`을 추가하고, 루프는 plan 체인을 따라간다(각 스테이지 종료 후 `!aborted`일 때만 다음으로, `"── 다음 단계 ──"` 로그, progress 스테이지마다 리셋, lines 이어붙임 — 배치 6 규칙 그대로). **연쇄 상한 5스테이지**(초과 시 system 경고 후 중단 — 무한 체인 방어). 기존 훅 인자 `nextStage`는 하위 호환으로 유지하되 내부적으로 첫 plan의 nextStage로 합류시킨다(둘 다 있으면 훅 인자가 우선하고 경고 로그 — 실제로는 공존 소비자 없음, 코드로 확인).
7. **건별 소요 시간 로그**: stepOne 결과 라인에 `(n.n초)` 접미(계측 문화 — 병목 후속 판단 재료).

### 2단계 — LLM 레이트리밋 status 보존 (`lib/llm/`)

`callLLM`·어댑터가 프로바이더 HTTP 오류를 어떻게 throw하는지 코드에서 확인한 뒤, **에러 객체에 status를 보존**한다(예: `LLMHttpError extends Error { status: number }` 또는 기존 에러에 `status` 프로퍼티 부여 — 기존 catch 경로 무영향 방식 선택). 메시지 문자열 파싱으로 429를 판별하는 방식은 금지(취약). 기존 호출부·safeError 관행 무영향을 build로 확인한다.

> retryable 판정을 **서버 액션이 수행**(status 429·503·529 → `retryable: true`)하는 배선은 배치 2·3에서 각 액션을 열 때 한다. 배치 1은 status 보존까지만.

### 3단계 — 테스트

훅의 풀·재시도·연쇄 핵심을 테스트 가능하게 만든다. 권장: 스케줄링 코어를 순수 함수/클래스(`lib/hooks/run-pool.ts` 등, React 비의존 — stepOne 실행기·플래그 리더를 주입받는 형태)로 추출하고 훅이 이를 감싼다. **신규 테스트 ≥ 10건**(package.json 추가): 동시성 1 = 직렬과 동일 순서 / 동시성 2~3에서 최대 in-flight 준수 / pause 후 신규 미투입·재개 / stop 시 in-flight 정산 후 1회 finalize·aborted / 연속 실패 3회 서킷(완료 순서 기준) / 성공이 카운터 리셋 / retryable 재시도 2회 후 최종 실패만 집계 / retryable 성공 시 failed 미집계 / 다단 연쇄 2회(3스테이지) 진행·progress 리셋 / 연쇄 상한 캡. 순수 추출이 구조상 과하다고 판단되면 **작업을 멈추고 사용자에게 대안(통합 수동 확인 목록)을 보고 후 승인받는다** — 임의 생략 금지.

## [배치 2 프롬프트] — 진실성 파이프라인 재편: 식별(ISBN 정규화) → 팩트시트 준비(책 단위) → 대조

`docs/리팩토링_3.md` 배치 2를 실행하라. 전제: 배치 1 완료. 범위: `evaluate/actions.ts`의 진실성 액션 재편 + `acquireFactsheet`류 확보 로직의 스테이지 분리. **phase2-panel 결선은 최소한**(동작 유지에 필요한 만큼만 — 동시성 UI·최종 결선은 배치 3). 채점(stage 채점)·`evaluateOne`은 건드리지 않는다.

### 설계 — 실행 흐름 (배치 3 완료 시점의 목표)

```text
[채점 실행] = 스테이지 연쇄 (useSequentialRun, 동시성 1~3)
  S1 식별   : 제출물 1건 단위 — URL 추출 + LLM SourceClaim + 알라딘 ISBN 정규화 → authenticity jsonb에 저장
  S2 팩트시트 준비 : 책 1권 단위 — isbn13(또는 제목 폴백 키)별 find-or-create + 부실 시 보강 1회
  S3 대조   : 제출물 1건 단위 — 준비된 팩트시트·URL 원문과 대조 → 플래그 저장 (생성 경로 없음)
  S4 채점   : 기존 prepareEvaluation / evaluateOne / finalizeEvaluation 그대로
```

- S2의 대상이 **고유 책**이므로 병렬이어도 `factsheets_owner_isbn` 충돌·이중 수집이 원천 차단된다(D-2).
- 모든 스테이지 증분: S1은 claim 없거나 content_hash 변경분만, S2는 미확보 책만, S3는 기존 증분 규칙(미검증 또는 해시 변경) 유지.
- 의심 판정은 여전히 플래그만(자동 감점·제외 없음 — 보수 원칙 불변).

### 1단계 — S1: 식별 + ISBN 정규화

`evaluate/actions.ts`에 신규(기존 `verifyAuthenticityOne`의 식별 부분을 이동·확장):

```ts
export async function prepareSourceIdentify(projectId: string):
  Promise<{ targets: { id: string; label: string }[]; skipped: number }>;
// 대상 = 채점 대상 조건과 동일한 제출물 중 authenticity.claim이 없거나 저장 content_hash ≠ 현재. LLM 없음.

export async function identifySourceOne(projectId: string, submissionId: string):
  Promise<{ ok: boolean; message: string; retryable?: boolean }>;
```

`identifySourceOne` 절차(throw 금지): ① 소속·자격·증분 재확인 ② `extractUrls` + `callLLM(purpose="추출")` → `parseSourceClaim` ③ kind=none·URL 0건 → `authenticity_status='not_applicable'` 저장, ok:true "출처 인용 없음" ④ **kind=book이고 알라딘 키 있으면 `searchBooks(claim.title[, author])` 최상위 후보들을 순회해 `metaAgrees`(strict-review의 순수 함수 — import 경계를 코드로 확인, server-only 파일이면 순수 함수를 공용 위치로 옮기되 strict-review 동작 불변) 느슨 일치하는 첫 후보의 isbn13을 확정**. 일치 후보 없거나 키 미설정이면 isbn13=null(제목 폴백) ⑤ 저장: `authenticity` jsonb에 `{ claim, isbn13, urls, content_hash, identified_at }` 병합(기존 findings 등 보존), status는 아직 unverified 유지 ⑥ LLM·알라딘 429/503/529는 `retryable:true`로 반환. message 예: `"『코스모스』 → ISBN 확정"` / `"『…』 — 알라딘 미등재(제목 매칭)"` / `"출처 인용 없음"`.

### 2단계 — S2: 팩트시트 준비 (책 단위)

```ts
export async function prepareFactsheetStage(projectId: string):
  Promise<{ targets: { id: string; label: string }[]; prelude: { level: "info" | "system"; text: string }[] }>;
// S1이 저장한 claim들에서 고유 책 키를 조립: isbn13 있으면 "isbn:{isbn13}", 없으면 "title:{정규화 제목}|{정규화 저자}".
// 이미 확보된 책(내 소유·shared에 isbn13 일치, 폴백 키는 정규화 제목+저자 일치 — 정규화 규칙은 공백 축약+trim, metaAgrees와 일관)은 제외.
// prelude: "준비 대상 책 N권 · 기확보 M권 재사용". 검색 키 미설정이면 targets 빈 배열 + system 안내(기능 비활성, 에러 아님).

export async function prepareFactsheetOne(projectId: string, bookKey: string):
  Promise<{ ok: boolean; message: string; retryable?: boolean }>;
// 책 1권: find(내 소유 isbn13 → shared isbn13 → 폴백 키) → 없으면 createFactsheetFromBook + 자동 수집(기존
// planCollection/collectOneSource, AUTH_COLLECT_MAX 유지) → 부실(entry<5)·내 소유면 보강 1회(기존 규칙).
// message 예: "『…』 재사용(entry 12)" / "『…』 자동 생성 — entry 7건" / "『…』 수집 실패(사유)".
```

기존 `acquireFactsheet`·`autoCollectFactsheet`의 로직을 이 액션으로 **이동**하고(중복 구현 금지), `verifyAuthenticityOne`에는 생성·수집 경로가 남지 않게 한다(아래 3단계). shared 우선 재사용·소유 팩트시트 우선순위 등 기존 규칙은 그대로 옮긴다.

### 3단계 — S3: 대조 전용으로 `verifyAuthenticityOne` 개편

- 팩트시트는 **조회만**: S1이 저장한 isbn13(없으면 폴백 키)으로 내 소유→shared 조회. 없으면(S2 실패건) 팩트시트 증거 없이 진행(대개 unverifiable — 기존 "근거 확보 실패" 경로).
- URL fetch(최대 3)·대조 LLM·`parseFindings`·`summarizeVerdict`·저장은 기존 그대로. LLM 429/503/529는 `retryable:true`.
- `prepareAuthenticity`는 S3 전용으로 유지(증분 규칙 불변, 이름 유지 — 소비처 phase2-panel 최소 수정).
- grep 확인: `verifyAuthenticityOne` 내부에 `searchBooks`·`createFactsheetFromBook`·`collectOneSource` 호출 잔존 0건.

### 4단계 — 최소 결선 + 테스트

- phase2-panel의 스테이지 연쇄를 S1→S2→S3→S4 4단으로 교체(배치 1의 plan 체인). 동시성 UI는 아직 넣지 않는다(기본 1 그대로 — 배치 3). 각 스테이지 finalize 요약: S1 "식별 — 도서 a·URL b·해당 없음 c·ISBN 확정 d" / S2 "팩트시트 — 재사용 a·생성 b·실패 c" / S3 기존 진실성 요약 유지.
- **신규 테스트**(순수 로직 분, package.json 추가, ≥ 8건): 책 키 조립(isbn 우선·폴백 정규화·중복 제거)·claim→키 매핑·(옮겼다면) metaAgrees 재배치 후 동작 불변·S1 jsonb 병합이 기존 findings를 보존하는 병합 함수. 서버 액션 자체는 기존 관행대로 수동 확인 목록.
