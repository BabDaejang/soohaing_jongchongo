## [공통 프롬프트] — 모든 배치 대화의 서두에 포함

너는 수행평가 수합·평가·생기부 생성 웹앱(**soohaing_jongchongo** — Next.js App Router + TypeScript strict + Tailwind + Supabase, 배포 Vercel) 저장소에서 `docs/리팩토링_2.md`의 **배치 하나만** 실행하는 개발자다.

### 시작 절차 (반드시 순서대로)

1. CLAUDE.md 세션 프로토콜대로 `docs/SPEC.md` → `docs/DATA_MODEL.md` → `docs/PROGRESS.md` → `docs/DECISIONS.md`를 읽는다.
2. `docs/리팩토링_2_프로그레스.md`를 읽고 체크리스트에서 **앞 배치가 완료됐는지**, 워킹 트리가 깨끗한지(`git status`) 확인한다. 미완이거나 기록과 실제 코드가 어긋나면 작업을 멈추고 사용자에게 보고한다.
3. `docs/리팩토링_2.md`에서 지시받은 배치 절만 정독한다. **다른 배치의 작업은 하지 않는다.**
4. 배치가 수정할 파일은 **수정 전에 반드시 전체를 읽는다**. 이 문서와 실제 코드가 어긋나면 **코드가 기준** — 확인 후 프로그레스 문서에 특이사항으로 기록한다.

### 배경 (요약 — 상세 근거는 각 배치 절)

- **PDF 버그 2건(2026-07-12 탐침 실증)**: ① `parsePdfText`(unpdf/pdf.js)가 입력 `Uint8Array`의 ArrayBuffer를 **detach**시켜(워커로 소유권 이전), 이후 스캔 PDF OCR 경로가 **빈 base64**를 전송 → Google `400 "The document has no pages"`. 어떤 모델을 골라도 실패한다(데이터 자체가 빈 것). ② OpenAI 형식은 PDF 파트를 무조건 throw하도록 하드코딩돼 있으나, 실제 OpenAI Chat Completions는 `file` 콘텐츠 파트로 PDF 입력을 지원한다.
- **UX 전면 개편(사용자 지시, 2026-07-12)**: 프로젝트 대시보드를 **페이즈 0(준비)→1(수합)→2(평가)→3(생기부)이 수직으로 이어지는 단일 작업 화면**으로 재구성하고, 하단에 **[작업결과표]**(8열 엑셀 시트형, 전폭)를 둔다. 기존 /ingest·/evaluate·/results·/students 라우트는 대시보드로 흡수·제거(사용자 확정). 매칭·확인(/submissions), 제출물 상세, 루브릭 편집(/rubric), 프롬프트 프로필(/profile), 생기부 문장 검증(/records)은 세부 화면으로 유지.
- **점수 체계(사용자 지시)**: 현행 루브릭 합산 점수는 폭이 좁아 상대 조정이 불가 → **999점 만점 표시 점수** 도입. 초기 15~25명 채점 완료까지 점수 미확정(표시 유보), 완료 시 800~200 범위에 최소 15점 간격으로 스프레드, 이후 학생은 기존 점수 사이 중간값으로 삽입. 루브릭 원점수·등급 파생(INV-6)은 유지된다.
- **진실성 검증 + 도서팩트시트(사용자 지시, 2026-07-13)**: 페이즈 2에서 **점수 부여에 앞서 제출물 진실성을 검증**한다. 독서 활동 제출물은 인용 도서를 특정해 **도서팩트시트**(메타·목차·소개 + 챕터별 내용, 전 항목 출처 필수)와 교차 대조하고, URL 인용(기사·웹페이지)은 원문을 직접 fetch해 대조한다 — AI 할루시네이션 제출물 검출 목적. 같은 책을 여러 학생이 고르므로 팩트시트는 저장·재사용해 토큰을 아낀다. 수집 채널은 **오픈 API 직접 수집**(알라딘 + 네이버, 관리자 환경변수 키 — 사용자 선택), 의심 판정은 **플래그만·채점은 진행**(사용자 선택), 공유는 **생성 계정 소유 → 관리자 승인 시 전체 공유(승인 전 AI 엄격 자동 재검증)**(사용자 확정). 팩트시트 자체를 할루시네이션으로 만드는 것은 절대 금지 — 모든 항목은 수집 원문 스니펫 대조를 통과해야 저장된다(구조적 강제).
- 실행 인프라(리팩토링 1 산출물)는 재사용한다: `useSequentialRun` 훅 + `RunTerminal`(1건 단위 실행·일시정지/재개/긴급중단·서킷 브레이커). **서버 상주 잡·폴링·잡 테이블 없음.** DB 변경은 **마이그레이션 0012(display_score)·0013(팩트시트·진실성) 두 개**뿐이다.

### 불변 규칙 (위반 금지)

- CLAUDE.md의 INV-1~6. 특히 `stepOne`류 서버 액션의 클라이언트 입력은 **id(또는 storage 경로·URL)뿐** — 내용·루브릭·명단은 서버가 DB에서 조립(INV-2). 등급은 계속 파생 계산만(INV-6): 999점 체계는 **표시 점수**를 바꾸는 것이지 등급 직접 입력이 아니다.
- **팩트시트 무할루시네이션 원칙(배치 8~11 전체)**: 팩트시트 entry는 **수집한 원문 텍스트에서 추출된 것만** 저장한다 — LLM 반환 항목의 인용 스니펫이 원문에 실제로 존재하는지 **서버가 문자열 대조로 강제**하고(`filterBySnippetMatch`), 불일치분은 폐기한다. LLM 내부 지식으로 챕터 내용을 생성하는 프롬프트·코드 경로를 만들지 않는다. 메타(목차·소개)는 알라딘 API 원본을 LLM 비경유로 저장한다. 진실성 '의심' 판정은 플래그와 근거 표시만 — **자동 감점·채점 제외·반영 해제 없음**(교사 판단, 보수 원칙).
- **비범위**: OpenAI responses API, 스트리밍, 서버 잡 큐, Supabase Realtime(행 갱신은 window 이벤트+refetch), 마이그레이션은 0012·0013 두 개만(0013은 배치 8에서 일괄), 루브릭 xlsx 재가져오기, NEIS 연동, 구글 커스텀서치(알라딘·네이버로 충분), 논문 전문 검색(URL·DOI가 명시된 것만 fetch 대조 — 제목만 인용된 논문은 unverifiable로 정직하게 표시), 크롤링 프레임워크(단순 fetch + 텍스트 추출만).
- **불가침(건드리지 말 것)**: `suggestMatchCandidates`(확인 큐 LLM 후보)·`reassignSubmission`(재귀속)·확인 큐 액션(attributeExisting/attributeNew/accept·rejectPendingContent)·`lib/crypto`·`lib/llm/keys.ts`·검증 패스(`parseVerification`)·등급 경계(`GRADE_BOUNDARIES`)·프롬프트 프로필 파이프라인(진입 위치만 옮김, 기능 불변).

```ts
    "2 0 obj\n<< /Type /Pages /Kids [3 0 R] /Count 1 >>\nendobj\n",
```

## [배치 2 프롬프트] — 999점 표시 점수 체계

`docs/리팩토링_2.md` 배치 2를 실행하라. 전제: 배치 1 완료. 범위: 표시 점수 순수 모듈 + 마이그레이션 0012 + 재계산 배치 개편 + override 검증 변경. **작업결과표·대시보드 UI는 하지 않는다**(배치 3~5).

### 설계 (사용자 지시 요약)

- 점수는 **999점 만점 정수 "표시 점수"**. 루브릭 채점(evaluations)·합성 원점수(composite)는 지금처럼 산출하되, 화면·조정에 쓰는 점수는 표시 점수다.
- **초기 확정 인원**: 채점 대상 학생 수(total)의 25%를 15~25명으로 클램프(대상이 15명 미만이면 전원). 이 인원이 채점되기 전에는 **표시 점수를 확정하지 않는다**(작업결과표 반영 점수 = 빈 칸). 원점수는 evaluations에 이미 저장되므로 "메모리에만 기록" 요구는 **student_scores 행을 만들지 않는 것**으로 실현한다(컬럼 nullable 변경 불필요).
- 인원 충족 시 원점수 순위대로 **800→200 등간격 스프레드**(15~25명이면 간격 25~43점 ≥ 15점 보장). 이후 채점되는 학생은 **기존 배정 학생들 사이 중간값**으로 삽입(최상위 진입 = 기존 최고와 999의 중간, 최하위 진입 = 기존 최저와 0의 중간). 정수 여유가 소진되면 **전체 재배치**(현재 원점수 순서로 재스프레드).
- 한 번 배정된 표시 점수는 **재계산에도 유지(sticky)** — 교사가 조정한 감각이 흔들리지 않게. 교사 오버라이드(`students.score_override`)는 이제 **0~999 정수**이며 표시 점수보다 우선한다(사유 필수 — DB check 제약 유지).
- 순위·등급은 `effective(= override ?? display)` 내림차순에서 파생(INV-6 불변, `computeStandings` 재사용).

### 2단계 — 순수 모듈 `lib/scores/display.ts` (신규)

```ts
export const DISPLAY_MAX = 999;
export const SPREAD_TOP = 800;
export const SPREAD_BOTTOM = 200;
export const MIN_INITIAL_GAP = 15;

// 초기 확정 인원: 대상 15명 미만이면 전원, 아니면 대상의 25%를 15~25로 클램프.
export function initialConfirmCount(totalTargets: number): number;

// n명을 SPREAD_TOP→SPREAD_BOTTOM 등간격 내림차순 배치(round 정수). n===1 → [500].
// n ≤ 41이면 간격 ≥ MIN_INITIAL_GAP이 자동 보장된다(600/40=15).
export function spreadScores(n: number): number[];

// 위/아래 이웃 표시 점수 사이의 삽입값. above 없음 → round((below+DISPLAY_MAX)/2),
// below 없음 → round(above/2), 둘 다 있으면 중간값(min/max 정규화 후). 정수 여유가 없으면 null.
export function insertBetween(above: number | null, below: number | null): number | null;

export type RawRankedStudent = { studentId: string; raw: number }; // 원점수 내림차순 정렬 입력(동점 허용)
export type AssignResult = {
  displays: Map<string, number>; // 전원 확정 시 rawRanked 전원의 표시 점수. 미확정 국면이면 빈 Map
  confirmed: boolean;            // false = 초기 확정 인원 미달(전부 미확정)
  respread: boolean;             // true = 공간 소진으로 전체 재배치가 일어남
};
export function assignDisplayScores(input: {
  rawRanked: RawRankedStudent[];          // 호출부가 원점수 내림차순 정렬 보장
  existing: ReadonlyMap<string, number>;  // 이전 배정(sticky). rawRanked에 없는 학생은 무시
  totalTargets: number;                   // 채점 대상 학생 총수(미채점 포함)
}): AssignResult;
```

`assignDisplayScores` 의미론(이 순서대로 구현):

1. `existing`에서 `rawRanked`에 없는 studentId 제거(탈락자 정리) → `kept`.
2. `kept`가 비어 있고 `rawRanked.length < initialConfirmCount(totalTargets)` → `{ displays: 빈 Map, confirmed: false, respread: false }`.
3. `kept`가 비어 있고 인원 충족 → **초기 스프레드**: `spreadScores(n)`을 rawRanked 순서대로 배정. **원점수 동점 그룹은 그룹 첫 위치의 값으로 통일**(같은 raw = 같은 display).
4. `kept`가 있으면: 기존 학생은 그대로 유지. 신규 학생을 raw 내림차순으로 하나씩 배치 — anchors = 이미 display를 가진 (raw, display) 집합(기존+이번에 배정된 신규 포함). ① 같은 raw의 anchor가 있으면 그 display 그대로(동점) ② above = raw가 더 큰 anchor들의 display 최솟값, below = raw가 더 작은 anchor들의 display 최댓값 → `insertBetween(above, below)` ③ null(소진)이면 **전량 재배치**: kept를 버리고 rawRanked 전체를 3번 규칙으로 재스프레드, `respread: true`. (재채점으로 기존 anchor의 raw 순서와 display 순서가 어긋난 경우에도 insertBetween의 min/max 정규화로 동작한다.)
5. confirmed=true면 displays는 rawRanked 전원을 포함해야 한다.

**신규 테스트 `tests/display-score.test.ts`** (package.json 추가, ≥ 14건): initialConfirmCount(3→3, 14→14, 15→15, 60→15, 100→25, 400→25) / spreadScores(1→[500], 2→[800,200], 25명 간격 25, 41명 간격 15, 등간격·내림차순) / 미달 시 빈 Map·confirmed=false / 초기 확정 동점 그룹 동일 점수 / 신규 중간 삽입 = 이웃 중간값 / 최상위 진입 = (최고+999)/2 / 최하위 진입 = 최저/2 / 신규 동점 = anchor와 동일 / 이웃 붙음(예: 415·416) → respread=true·순서 보존·전원 배정 / existing의 탈락자 제거 / 같은 입력 2회 → 동일 출력(결정성).

### 3단계 — `recomputeAndSave` 개편 (app/projects/[id]/evaluate/actions.ts)

기존 "삭제 → 재삽입" 구조를 유지하되 표시 점수를 끼워 넣는다:

1. 삭제 **전에** 기존 배정을 조회: `admin.from("student_scores").select("student_id, display_score")` → `existing` Map(값이 null인 행은 제외).
2. 원점수 계산(기존 로직 그대로: perStudent → composite → effective는 아래에서 재정의).
3. `totalTargets` = 채점 대상 학생 수 = subToStudent 값(학생 id)의 고유 개수(제출물이 있는 학생 전원 — 아직 채점 안 된 학생 포함). **override만 있고 제출물 평가가 없는 학생은 totalTargets에 넣지 않는다.**
4. `rawRanked` = rows(평가 있는 학생)를 composite 내림차순 정렬(동률은 studentId 사전순 — 결정성).
5. `assignDisplayScores({ rawRanked, existing, totalTargets })` 호출.
6. **confirmed=false(미확정 국면)**: student_scores를 전부 삭제만 하고 삽입하지 않는다. needs_recalc 해제는 기존대로. 반환값에 미확정 정보를 담는다(아래 7).
7. **confirmed=true**: 각 row에 대해 `display = displays.get(studentId)`, `effective = override ?? display`(override는 이제 999 스케일). override만 있고 평가 없는 학생은 display=null·effective=override로 포함(랭킹 대상 기존 규칙 유지). `computeStandings(effective 배열)`로 rank·grade → insert 컬럼에 `display_score` 추가.
8. 반환 타입을 `{ ranked: number; pendingConfirm: { scored: number; required: number } | null }`로 변경 — 호출부 3곳(`finalizeEvaluation`·`recalculate`·`setScoreOverride`/`clearScoreOverride`)을 갱신: finalize/recalculate는 미확정이면 로그·메시지를 "채점 {scored}/{required}명 — 표시 점수 확정 대기(최소 {required}명 채점 후 확정)"로, 확정이면 기존 "재계산 — N명 순위 산출".

### 4단계 — override 검증·표시 소폭 수정

- `setScoreOverride`: 검증을 **정수 0~999**로 변경(`Number.isInteger(value) && value >= 0 && value <= 999`, 위반 메시지 "보정 점수는 0~999 정수입니다."). 사유 필수·감사 로그·재계산 호출은 불변.
- `evaluateOne` 성공 메시지 `` `${total}점` `` → `` `원점수 ${total}점` ``(확정 표시 점수는 재계산 후 산출됨을 로그에서 구분).
- `components/projects/evaluate/results-table.tsx`(현행 평가 화면 학생 표 — 파일을 열어 구조 확인): 점수 열을 표시 점수 기준으로 — `display_score`(또는 override 시 "999점 스케일 보정 값 + 배지") 우선 표시, composite는 "원점수 n" 부가 표기. 이 표는 배치 7에서 작업결과표로 대체·제거될 예정이므로 **최소 수정**만 한다. run-panel의 "채점 대상 …" 안내문에 한 문장 추가: "표시 점수는 초기 확정 인원(15~25명) 채점 후 999점 만점으로 확정됩니다."
- 페이지가 student_scores를 select하는 곳에 `display_score` 컬럼 추가(`lib/supabase/types.ts`의 student_scores Row 타입에 `display_score: number | null` 추가 — 수동 관리 타입임).

```ts
// contradicted ≥ 1 → suspect / 주장 ≥ 1이고 supported 비율 ≥ 0.5이며 contradicted 0 → verified / 그 외(전부 not_found 등) → unverifiable
```

## 부록 — 전체 수용 기준 (사용자 요구 ↔ 검증)

1. 통사 9반 PDF: 업로드 → 수합 실행 시 "The document has no pages" **재발 없음**, OpenAI 모델 선택 시에도 PDF OCR 동작(배치 1 + 배포 후 확인).
2. 업로드 즉시 토큰 소모 없음 — [수합 & 매칭] 클릭 시에만 추출·매칭 실행, 업로드 파일 확인·체크 제외·삭제 가능(배치 6).
3. 수합 완료 후 매칭이 같은 터미널에서 자동 연쇄, 애매 건만 확인 대기 큐(배치 6).
4. OCR 모델이 프로바이더 저장 목록 기반 드롭다운(비전 가능 필터), gpt-4o만 고정 제공되던 제약 소멸(배치 1·6).
5. 점수 999점 만점: 초기 15~25명 확정 대기 → 800~200 스프레드(간격 ≥ 15) → 사이 삽입 → 소진 시 재배치, 배정 점수는 재계산에도 유지(배치 2).
6. 대시보드 = 페이즈 0~3 수직 + 하단 작업결과표(전폭), 각 페이즈 [작업결과표 바로보기], 빈 칸 클릭 → 해당 페이즈 스크롤(배치 3~5).
7. 페이즈 0: API 키 상태·인라인 등록·기본 AI 모델·전 페이즈 교체 질문(배치 5).
8. 공용 실행 터미널이 페이즈 1·2·3 전부에 상시 표시, 상단 배치, 1명 처리마다 작업결과표 즉시 갱신(배치 6·7).
9. 작업결과표 8열 + 정렬·필터·감춤·열너비·행높이·전체 접기/펼치기 + 제출물 갯수 펼침 + 오버라이드(저장/취소·사유) + 학생별 데이터 편집(+추가/삭제 — 학생 명단 대체) + xlsx/csv/md 다운로드(전체/선택, `프로젝트명-yymmddHHmm`)(배치 3·4).
10. 루브릭 = 페이즈 2 하위: 전용 편집 화면 + 평가계획서 업로드 → AI 제안 → 교사 승인 반영 + 전담 모델 + 다운로드(배치 7).
11. 프로필 = 페이즈 3 하위 + 생성 모델 가성비 추천, 생기부 일괄 생성 터미널(INV-1 유지)(배치 7).
12. 채점 실행은 진실성 검증 스테이지가 선행: 인용 출처 식별 → 도서는 팩트시트, URL은 원문과 대조 → 확인/의심/판정 불가/해당 없음 플래그 + 근거 인용 표시, **자동 감점·채점 제외 없음**(배치 10).
13. 제출물의 도서가 팩트시트에 없으면 실행 중 자동 생성(알라딘 메타·목차 + 웹 원문 entry), 같은 책 2번째 학생부터 재사용 — 신규 수집 없이 대조(토큰 절약)(배치 9·10).
14. 팩트시트 entry는 수집 원문 발췌가 실존하는 것만 저장(할루시네이션 구조 차단), 전용 페이지에서 검색·생성·자동 보강(중복 방지·새 챕터 우선)·수동 편집·촬영본 OCR 보강(배치 8·9).
15. 공유는 관리자 승인 경유: 교사 신청 → 관리자 AI 엄격 검증(출처 재수집·발췌 실존 대조 리포트) → 승인 시 전 계정 재사용, shared는 읽기 전용(복제 후 보강)(배치 11).
16. `npm test`·typecheck·lint·build 통과, INV-4 번들 스캔 9종 0건, INV-1~6 위반 없음(배치 12).
