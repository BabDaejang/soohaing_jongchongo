## 0. 목적

대한민국 고등학교 교사가 학생 수행평가 산출물을 수합·평가하고, 학교생활기록부(생기부) 서술을 생성하는 웹 애플리케이션.

**최우선 요구사항 두 가지:**

1. **학생 간 데이터 혼입 절대 금지**
2. **산출물에 근거하지 않은 서술(할루시네이션) 금지**

## 1. 기술 스택

- **프론트/백엔드**: Next.js (App Router, TypeScript strict), Tailwind CSS
- **인프라**: Supabase — Auth, Postgres(RLS 필수), Storage(임시 원본 보관용)
- **배포**: Vercel
- **LLM 호출**: 전부 서버 측(Route Handler / Server Action)에서만 수행. 클라이언트에 API 키 노출 금지.
- **파일 파싱**:
  - xlsx/csv: SheetJS · papaparse
  - docx: mammoth
  - pdf: 텍스트 레이어 추출 + 스캔본은 LLM 비전 OCR (등록된 API 키의 비전 모델 사용, 외부 OCR 서비스 불사용)

## 2. 사용자·권한·인증

- **역할**: `admin`(관리자, 최초 1인 = 개발 의뢰자), `user`(교사)
- **인증 방식**: Google OAuth (Supabase Auth의 Google 프로바이더). 요청 스코프는 기본(email, profile)만 사용한다 — 민감 스코프가 없어 Google 앱 검증이 불필요하다.
- **가입 흐름**: 최초 Google 로그인 = 가입. `profiles` 레코드 `status='pending'` 자동 생성 → 관리자가 승인/거부/삭제.
- **미승인 사용자**: 로그인 시 대기 화면만 표시. 대기 화면 안내문은 관리자가 편집 가능(`app_settings.waiting_message`).
- **접근 제어**: 승인된 사용자만 앱 기능 접근 가능. 미들웨어에서 강제.

## 3. API 키 체계

- **프로바이더**: 기본 시드 `google` / `anthropic` / `openai`. 관리자 화면 하단 [+] 버튼으로 프로바이더 추가 가능(name, base_url, API 형식: anthropic / openai호환 / google).
- **기본 키**: 관리자가 프로바이더별로 등록. 개인 키를 등록하지 않은 사용자에게 적용.
- **개인 키**: 각 사용자가 계정 옵션에서 등록/변경/삭제. 존재하면 기본 키보다 우선.
- **저장**: AES-256-GCM으로 암호화(키는 환경변수 `APP_ENCRYPTION_KEY`). 화면에는 끝 4자리만 마스킹 표시. 평문 저장·로그 출력 금지.
- **키 검증**: 키 등록/변경 시 서버가 해당 프로바이더의 모델 목록 엔드포인트를 호출해 유효성을 확인한다. 호출에 실패하면 키를 저장하지 않고 사유를 화면에 표시한다.
- **모델 목록**: 검증에 성공하면 조회된 모델 ID 목록을 키와 함께 저장한다. 모델은 프로바이더 사정으로 바뀌므로 키별 [모델 갱신] 버튼으로 재조회할 수 있고, 마지막 조회 시각을 표시한다.
- **모델 라우팅**: 프로젝트 설정에서 용도별 모델 지정 — 추출/매칭용(저비용), 평가용, 생성용, 검증용.
  - 기본값: 추출/매칭 = `claude-haiku-4-5`, 평가/생성/검증 = `claude-sonnet-4-6`.
  - **프로바이더 선택**: 사용자가 쓸 수 있는 키(개인 키 우선, 없으면 관리자 기본 키)가 등록된 프로바이더만 목록에 나온다. 쓸 수 있는 키가 하나도 없으면 전 프로바이더를 비활성(회색)으로 표시하고 계정 옵션에서 키를 등록하라는 안내를 띄운다.
  - **모델 선택**: 해당 키에 저장된 모델 목록에서 고른다. 목록에 없는 모델은 직접 입력할 수 있다(커스텀 프로바이더 대응).

## 4. 프로젝트 구조

- 로그인 후 첫 화면: 프로젝트 목록(생성/수정/삭제). **프로젝트 = 하나의 수행평가 단위.**
- **프로젝트 대시보드**: 프로젝트 선택 시 프로젝트 대시보드로 진입한다. 대시보드는 페이즈 0(준비: API 키 상태·기본 AI 모델·설정) → 1(수합) → 2(평가) → 3(생기부) 섹션이 수직으로 배열된 단일 작업 화면이며, 최하단에 **작업결과표**가 전폭으로 표시된다. 각 페이즈에 [작업결과표 바로보기]가 있고, 작업결과표의 빈 칸을 누르면 해당 작업 페이즈로 스크롤한다.
- **프로젝트 설정 항목**:
  - 등급제(5등급/9등급)
  - 생기부 글자수 제한 및 카운트 방식(글자수/바이트) — 기본값 500자, 글자수(공백 포함)
  - 평가 루브릭
  - 모델 라우팅
  - 기본 AI 모델(default 라우팅) — 각 용도 선택의 기본값. 변경 시 개별 설정을 일괄 교체할지 질문
  - 원본 파일 삭제 정책
  - 합성 점수 방식(합/평균/가중) — 기본값 평균
  - 동점자 처리(상위 등급 부여/중간석차) — 기본값 상위 등급 부여

## 5. Phase 1 — 수행평가 데이터 수합

### 5.1 업로드

- 다중 파일 업로드: xlsx, csv, docx, pdf(텍스트/스캔), 이미지.
- **업로드와 추출 분리**: 업로드는 원본 저장까지만 수행하고 토큰을 소모하지 않는다. 업로드된 파일 목록(체크박스 기본 선택, 미수합 파일 삭제 가능)을 교사가 확인한 뒤 [수합 & 매칭] 버튼으로 추출을 실행한다. 실행은 파일 1건 단위로 진행 터미널에 표시되고 일시정지·재개·긴급 중단할 수 있다.
- **OCR 모델**: OCR 모델은 등록된 키의 저장 모델 중 비전 가능 모델을 드롭다운으로 고른다(커스텀은 직접 입력). 스캔 PDF는 anthropic·google·openai(file 파트) 모두 지원.
- **구글 스프레드시트**: 교사가 xlsx(또는 CSV)로 내보내 업로드한다. 별도 API 연동 없음.
- **재업로드 중복·변경 감지**: 같은 시트를 갱신해 반복 재업로드하는 사용 패턴을 전제로, 업로드 시 기존 제출물과의 중복·변경을 감지한다:
  - (학생 식별값 + 제출물ID)가 동일하고 내용 지문(정규화 텍스트 해시)도 같으면 → 건너뛴다.
  - 내용이 다르면 → 갱신 확인 큐에 올린다. **자동 덮어쓰기 금지.**
- **열 매핑 UI**: 스프레드시트는 학번/이름/제출물ID/내용 열을 교사가 지정. LLM이 헤더를 보고 초기 추천, 교사가 확정.

### 5.2 학생 식별·매칭 규칙 (결정적 규칙 우선, LLM은 보조)

**식별값 확보** — 매칭에 쓰는 학번·이름은 세 경로로 얻고, 출처를 `identity_source`에 기록한다.

- `column` — 스프레드시트의 학번/이름 열 (교사가 열 매핑에서 확정).
- `filename` — 개별 파일(docx/pdf/이미지)의 파일명을 **학생 명단과 교차 대조**해 얻는다. 명단에 실재하는 학번·이름이 파일명에 온전한 토큰으로 들어 있을 때만 인정한다(예: `수행평가최종.docx`의 "수행평가"는 명단에 없으므로 무시). 명단의 서로 다른 학생 둘 이상이 걸리면 인정하지 않는다.
- `llm` — 위 둘이 모두 실패하면 문서 앞부분을 저비용 모델로 읽어 **명단의 학생 한 명**을 지목하게 한다. 명단에 없는 응답은 버린다(환각 방지).

**귀속 판정** — 확보한 식별값으로 다음을 적용한다.

- (a) **학번이 명단과 완전 일치** → 자동 귀속 (`auto_number`)
- (b) **이름이 명단에 정확히 1명만 일치** → 자동 귀속 (`auto_name`). 2명 이상 일치(동명이인)하거나 0명이면 확인 대기 큐.
- (c) **학번이 있으나 명단에 없음**: 출처가 `column`이고 이름 충돌이 없으면 신규 학생 자동 생성(아래 d). 그 외(파일명·LLM 유래)는 확인 대기 큐 — 오타 하나로 유령 학생이 생기는 것을 막는다.
- (d) **신규 학생 자동 생성은 `column` 출처의 신규 학번에 한한다.** 그 학번의 이름이 명단의 다른 학생과 일치하면(학번 오타 의심) 자동 생성하지 않고 확인 대기 큐로 보낸다.
- (e) **식별값 미확보** → 확인 대기 큐. LLM 후보 제안은 큐에서 지연 실행하고, 반드시 교사 확인 후 귀속한다.

**확인 대기 큐에는 애매한 것만 남긴다.** 동명이인·명단 미일치·학번 충돌·식별 불가·재업로드 내용 변경이 그것이다.

**매칭 실행은 1건 단위로 수행한다.** 결정적 규칙(열·파일명 교차 대조)은 실행 시작 시 한 번에 처리하고, LLM 추출이 필요한 건만 진행 터미널로 건별 로그를 표시하며 일시정지·재개·긴급 중단할 수 있다. 중단 시 처리분까지 반영되고, 재실행하면 미매칭 건만 증분으로 이어간다. 수합 실행이 끝나면 같은 터미널에서 매칭이 자동으로 이어진다(동명이인·식별 불가는 종전대로 확인 대기 큐).

**자동 귀속은 되돌릴 수 있어야 한다.** 학생별 제출물 화면에서 제출물을 다른 학생으로 **재귀속**할 수 있고(`reassigned`), 감사 로그에 이전/이후 학생을 남긴다. 귀속 경로(`match_method`)와 식별값 출처(`identity_source`)를 화면에 배지로 표시해 교사가 자동 처리 결과를 훑어볼 수 있게 한다.

### 5.3 저장 정책

- DB에는 **추출·정제된 텍스트만** 저장(Supabase 용량 제약).
- 원본 파일은 Storage 임시 버킷에 두고, 교사가 추출 텍스트를 확인(추출 품질 승인)한 뒤 삭제한다. **확인 전 삭제 금지(INV-5).**
- 프로젝트 설정으로 "N일 후 자동 삭제" 보조 정책 선택 가능 — 기본값 끄기, 선택지 7일/30일

### 5.4 제출물 상세 화면

- 학생별 제출물 목록·내용 열람.
- 제출물마다 평가/생기부 반영 체크박스(기본 체크됨).
- 수정/삭제/수동 추가 가능.
- **재귀속**: 잘못 귀속된 제출물을 다른 학생으로 옮길 수 있다. 자동 귀속(5.2)의 오류를 교사가 사후에 바로잡는 경로이며, 감사 로그와 "재계산 필요" 배지를 남긴다.

### 5.5 반복 수합

- 언제든 Phase 2·3 진행 중에도 Phase 1으로 돌아와 자료 추가 가능. 추가 시 매칭 규칙 동일 적용.

## 6. Phase 2 — 상대평가 등급 부여

- **실행 시점**: 교사가 결정(버튼). 신규 제출물·체크박스 변경 발생 시 "재계산 필요" 배지 표시. 채점 실행은 진행 터미널로 건별 로그를 표시하며, 일시정지·재개·긴급 중단할 수 있다. 중단 시 처리분까지 반영되고, 재실행하면 증분으로 이어간다.
- **파이프라인**:
  1. 제출물 단위 LLM 채점 (루브릭 기반, temperature를 지원하는 모델은 0 고정(미지원 추론 모델은 모델 기본값), 기준별 근거 인용 필수, 원문 출력 감사용 보관)
  2. 학생별 합성 점수 (합/평균/가중 중 프로젝트 설정)
  3. 합성 점수로 전체 순위
  4. 누적 비율로 등급 파생
- **표시 점수**: 합성 원점수는 순위 산출용이며, 화면·조정에는 **999점 만점 표시 점수**를 쓴다. 초기 확정 인원(채점 대상의 25%를 15~25명으로 클램프, 15명 미만이면 전원)이 채점되기 전에는 표시 점수를 확정하지 않고, 충족 시 원점수 순위대로 800~200 범위 등간격(최소 15점)으로 배정한다. 이후 채점 학생은 기존 배정 사이의 중간값으로 삽입하고, 정수 여유가 소진되면 순서를 보존해 전체 재배치한다. 배정된 표시 점수는 재계산에도 유지된다. 교사 보정(score_override)은 0~999 정수(사유 필수)로 표시 점수보다 우선한다. 등급은 표시 점수 순위에서 파생 계산한다(INV-6 불변).
- **등급 매핑**:
  - 5등급 = 누적 10 / 34 / 66 / 90 / 100 (%)
  - 9등급(스테나인) = 누적 4 / 11 / 23 / 40 / 60 / 77 / 89 / 96 / 100 (%)
  - 프로젝트 설정에서 토글, 화면 즉시 반영.
- **동점자 처리**: 프로젝트 설정 (동점자 상위 등급 부여 / 중간석차 방식).
- **수동 조정**: 원 명세의 "점수를 수동으로 조정해 순위를 바꾸는" 방식은 **채택하지 않는다**. 대신:
  - 점수는 항상 제출물 채점에서 파생 재계산한다.
  - 교사 개입이 필요하면 학생별 `score_override` 필드를 둔다 (사유 필수, 감사 로그).
  - 등급을 직접 저장·수정하는 것은 금지(INV-6).
- **루브릭**: 루브릭 편집은 Phase 2의 하위 행동이다. 평가계획서 파일을 업로드하면 전담 모델(라우팅 `rubric ?? default ?? evaluate`)이 기준·배점을 제안하고, 교사가 승인한 항목만 반영된다(자동 반영 금지). 루브릭은 xlsx로 내보낼 수 있다.

### 6.1 진실성 검증

채점 실행은 **진실성 검증 스테이지가 선행**된다(같은 진행 터미널에서 연쇄). 제출물이 인용한 출처(도서·기사·웹페이지·논문)를 식별해, 도서는 도서팩트시트와, URL은 원문 fetch 결과와 문장 단위로 대조하고 근거(supported/contradicted/not_found + 인용)를 남긴다. 판정은 확인/의심/판정 불가/해당 없음 4상태의 **플래그와 근거 표시만** 하며, 자동 감점·채점 제외는 하지 않는다(조치는 교사 판단 — 반영 체크 해제·점수 보정 등 기존 수단). 제출물의 도서가 팩트시트에 없으면 실행 중 자동 생성한다. 내용이 바뀌지 않은 제출물은 재검증하지 않는다(증분).

### 6.2 도서팩트시트

도서팩트시트는 도서의 메타(제목·저자·출판사·출간연도·목차·소개)와 **챕터별 내용 항목**(각 항목에 출처 URL·원문 발췌 필수)의 저장소다. 같은 책을 고른 여러 학생의 교차 검증에 재사용해 토큰을 아낀다. 수집은 오픈 API(알라딘: 메타·목차 / 네이버 검색: 웹 문서 URL) + 원문 fetch로 하며, **LLM은 수집 원문에서 추출·정제만** 하고 발췌 스니펫이 원문에 실존하는 항목만 저장된다(할루시네이션 구조 차단). 검색 API 키는 관리자가 환경변수(ALADIN_TTB_KEY·NAVER_CLIENT_ID/SECRET)로 1회 등록한다. 전용 페이지(/factsheets)에서 도서 검색·생성, 항목 추가/수정/삭제, 자동 보강(부실 시 새 챕터 우선·중복 방지), 책 페이지 촬영본(사진·캡처 PDF) 업로드 OCR 보강을 제공한다. **공유 모델**: 새 팩트시트는 생성 계정 소유이며, 전체 공유는 교사의 신청 후 **관리자 승인**을 거친다 — 관리자 화면의 AI 엄격 자동 검증(출처 재수집·발췌 실존 대조·내용 뒷받침 판정 리포트)을 참고해 승인/반려하고, 공유된 팩트시트는 읽기 전용(보강하려면 내 계정으로 복제)이다.

## 7. Phase 3 — 생기부 작성 (핵심 과제)

### 7.1 서술 원칙

- 산출물 내용 요약이 아니라 **학생의 경험 중심**: 동기 → 과정·노력·사고 과정 → 결과 → 배움·변화.
- 산출물과 교사 메모에 **근거가 있는 내용만** 서술.

### 7.2 생성 격리 (혼입 방지의 구조적 보장)

- **INV-1**: 생기부 생성 LLM 호출 1회에는 정확히 한 명의 학생 데이터만 포함된다. 일괄 생성도 내부적으로는 학생별 순차 호출이다.
- **INV-2**: 생성 컨텍스트는 서버가 `student_id` 필터로 DB에서 직접 조립한다. 클라이언트가 임의 텍스트를 주입할 수 없다 (교사 메모 필드는 예외이며 해당 학생 레코드에 귀속된 것만).
- **INV-3**: 생성 결과에는 근거로 사용한 제출물 id 목록(sources)을 함께 저장한다.

### 7.3 검증 패스

- 생성 직후 별도 LLM 호출(검증용 모델)로 초안을 문장 단위 분해 → 각 문장이 해당 학생의 제출물/교사 메모에 근거하는지 판정.
- 근거 없는 문장은 UI에서 하이라이트. 교사는 해당 문장 삭제/수정/재생성 선택.

### 7.4 교사 개인 관찰 메모

- 학생 정보 옆에 상시 노출되는 입력 박스. 생성 시 컨텍스트에 포함.

### 7.5 프롬프트 프로필 (생기부 메타데이터)

- 프롬프트 프로필 진입은 Phase 3(생기부)의 하위 배치다. 생성 모델이 미설정이면 등록 키의 모델 중 가격 대비 성능 우선순위로 추천해 보여준다.
- "작성 참고사항" 목록 + "금지사항" 목록.
- 화면은 좌(참고)/우(금지) 2패널, 사이 분할바 드래그로 폭 조절. 항목 추가/수정/삭제 가능.
- **프로필 계층**: 계정 기본 프로필(모든 프로젝트에 적용) + 프로젝트별 오버라이드. 계정별로 따로 관리·저장.
- **예시 생기부 인제스트(프롬프트 업그레이드)**: 교사가 좋은 생기부 예시 텍스트를 입력 → LLM이 분석해 참고/금지 항목 추가·수정안을 **diff 형태로 제안** → 교사가 승인한 항목만 반영. **자동 반영 금지.**
- **문체 기본값(시드 프로필에 포함)**: 종결어미 '-함/-임/-됨', 학생 성명·인칭대명사 미표기 등.
- **버전·이력·MD 입출력 (2026-07-10 추가, 사용자 승인)**: 프로필을 저장/예시반영/가져오기/복원할 때마다 **버전 번호를 올리고 업데이트 날짜시간을 기록**해 화면에 표시한다. 현재 프로필을 버전·날짜 헤더가 붙은 **Markdown으로 내보내기**하고, 편집한 Markdown을 **미리보기 후 가져오기**할 수 있다. **과거 버전 열람·복원**을 제공한다(이력은 append-only, 복원=새 버전 저장). 근거: DECISIONS 2026-07-10.

### 7.6 글자수 카운터

- 프로젝트 설정의 제한값 대비 실시간 표시.
- 카운트 방식 토글: 글자수(공백 포함) / 바이트(한글 3바이트).

## 8. 작업결과표

대시보드 하단의 전폭 엑셀 시트형 표. 열: ID(제품 부여 고유 번호)·학번·이름·업로드 된 제출물 갯수·반영 점수·등급·생성된 생기부·교사 관찰 메모. 1행 헤더에서 정렬·값 필터·열 감추기, 열 너비·행 높이 조절, 전체 접기/펼치기. 제출물 갯수를 누르면 제출물 제목 목록이 펼쳐진다. 반영 점수는 저장/취소 버튼이 있는 오버라이드 편집, 학번·이름은 [학생별 데이터 편집](ID 제외 대부분 수정 가능·학생 추가/삭제 포함 — 학생 명단 화면을 대체), 생기부·메모는 인라인 편집. 각 페이즈 실행에서 학생 1명 처리가 끝날 때마다 행이 즉시 갱신된다. xlsx/csv/md 다운로드(전체/선택, 파일명 `프로젝트명-yymmddHHmm`). 레이아웃은 (사용자, 프로젝트) 단위 저장·복원.
