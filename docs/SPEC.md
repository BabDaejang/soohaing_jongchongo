## 0. 목적

대한민국 고등학교 교사가 학생 수행평가 산출물을 수합·평가하고, 학교생활기록부(생기부) 서술을 생성하는 웹 애플리케이션.

**최우선 요구사항 두 가지:**

1. **학생 간 데이터 혼입 절대 금지**
2. **산출물에 근거하지 않은 서술(할루시네이션) 금지**

## 1. 기술 스택

- **프론트/백엔드**: Next.js (App Router, TypeScript strict), Tailwind CSS
- **인프라**: Supabase — Auth, Postgres(RLS 필수), Storage(임시 원본 보관용)
- **배포**: Vercel
- **LLM 호출**: 전부 서버 측(Route Handler / Server Action)에서만 수행. 클라이언트에 API 키 노출 금지.
- **파일 파싱**:
  - xlsx/csv: SheetJS · papaparse
  - docx: mammoth
  - pdf: 텍스트 레이어 추출 + 스캔본은 LLM 비전 OCR (등록된 API 키의 비전 모델 사용, 외부 OCR 서비스 불사용)

## 2. 사용자·권한·인증

- **역할**: `admin`(관리자, 최초 1인 = 개발 의뢰자), `user`(교사)
- **인증 방식**: Google OAuth (Supabase Auth의 Google 프로바이더). 요청 스코프는 기본(email, profile)만 사용한다 — 민감 스코프가 없어 Google 앱 검증이 불필요하다.
- **가입 흐름**: 최초 Google 로그인 = 가입. `profiles` 레코드 `status='pending'` 자동 생성 → 관리자가 승인/거부/삭제.
- **미승인 사용자**: 로그인 시 대기 화면만 표시. 대기 화면 안내문은 관리자가 편집 가능(`app_settings.waiting_message`).
- **접근 제어**: 승인된 사용자만 앱 기능 접근 가능. 미들웨어에서 강제.

## 3. API 키 체계

- **프로바이더**: 기본 시드 `google` / `anthropic` / `openai`. 관리자 화면 하단 [+] 버튼으로 프로바이더 추가 가능(name, base_url, API 형식: anthropic / openai호환 / google).
- **기본 키**: 관리자가 프로바이더별로 등록. 개인 키를 등록하지 않은 사용자에게 적용.
- **개인 키**: 각 사용자가 계정 옵션에서 등록/변경/삭제. 존재하면 기본 키보다 우선.
- **저장**: AES-256-GCM으로 암호화(키는 환경변수 `APP_ENCRYPTION_KEY`). 화면에는 끝 4자리만 마스킹 표시. 평문 저장·로그 출력 금지.
- **키 검증**: 키 등록/변경 시 서버가 해당 프로바이더의 모델 목록 엔드포인트를 호출해 유효성을 확인한다. 호출에 실패하면 키를 저장하지 않고 사유를 화면에 표시한다.
- **모델 목록**: 검증에 성공하면 조회된 모델 ID 목록을 키와 함께 저장한다. 모델은 프로바이더 사정으로 바뀌므로 키별 [모델 갱신] 버튼으로 재조회할 수 있고, 마지막 조회 시각을 표시한다.
- **모델 라우팅**: 프로젝트 설정에서 용도별 모델 지정 — 추출/매칭용(저비용), 평가용, 생성용, 검증용.
  - 기본값: 추출/매칭 = `claude-haiku-4-5`, 평가/생성/검증 = `claude-sonnet-4-6`.
  - **프로바이더 선택**: 사용자가 쓸 수 있는 키(개인 키 우선, 없으면 관리자 기본 키)가 등록된 프로바이더만 목록에 나온다. 쓸 수 있는 키가 하나도 없으면 전 프로바이더를 비활성(회색)으로 표시하고 계정 옵션에서 키를 등록하라는 안내를 띄운다.
  - **모델 선택**: 해당 키에 저장된 모델 목록에서 고른다. 목록에 없는 모델은 직접 입력할 수 있다(커스텀 프로바이더 대응).

## 4. 프로젝트 구조

- 로그인 후 첫 화면: 프로젝트 목록(생성/수정/삭제). **프로젝트 = 하나의 수행평가 단위.**
- **프로젝트 설정 항목**:
  - 등급제(5등급/9등급)
  - 생기부 글자수 제한 및 카운트 방식(글자수/바이트) — 기본값 500자, 글자수(공백 포함)
  - 평가 루브릭
  - 모델 라우팅
  - 원본 파일 삭제 정책
  - 합성 점수 방식(합/평균/가중) — 기본값 평균
  - 동점자 처리(상위 등급 부여/중간석차) — 기본값 상위 등급 부여

## 5. Phase 1 — 수행평가 데이터 수합

### 5.1 업로드

- 다중 파일 업로드: xlsx, csv, docx, pdf(텍스트/스캔), 이미지.
- **구글 스프레드시트**: 교사가 xlsx(또는 CSV)로 내보내 업로드한다. 별도 API 연동 없음.
- **재업로드 중복·변경 감지**: 같은 시트를 갱신해 반복 재업로드하는 사용 패턴을 전제로, 업로드 시 기존 제출물과의 중복·변경을 감지한다:
  - (학생 식별값 + 제출물ID)가 동일하고 내용 지문(정규화 텍스트 해시)도 같으면 → 건너뛴다.
  - 내용이 다르면 → 갱신 확인 큐에 올린다. **자동 덮어쓰기 금지.**
- **열 매핑 UI**: 스프레드시트는 학번/이름/제출물ID/내용 열을 교사가 지정. LLM이 헤더를 보고 초기 추천, 교사가 확정.

### 5.2 학생 식별·매칭 규칙 (결정적 규칙 우선, LLM은 보조)

- (a) **학번 완전 일치** → 기존 학생에 자동 귀속
- (b) **이름만 일치(학번 미검출)** → 자동 귀속 금지, "확인 대기 큐"에 넣고 교사가 확정 (동명이인 보호)
- (c) **식별 정보 미검출** → LLM이 문서 내용에서 후보를 제안하되 반드시 교사 확인 후 귀속
- (d) **신규 학생 여부 판단도 동일**: 자동 생성은 학번 신규 검출 시에만, 그 외는 교사 확인

### 5.3 저장 정책

- DB에는 **추출·정제된 텍스트만** 저장(Supabase 용량 제약).
- 원본 파일은 Storage 임시 버킷에 두고, 교사가 추출 텍스트를 확인(추출 품질 승인)한 뒤 삭제한다. **확인 전 삭제 금지(INV-5).**
- 프로젝트 설정으로 "N일 후 자동 삭제" 보조 정책 선택 가능 — 기본값 끄기, 선택지 7일/30일

### 5.4 제출물 상세 화면

- 학생별 제출물 목록·내용 열람.
- 제출물마다 평가/생기부 반영 체크박스(기본 체크됨).
- 수정/삭제/수동 추가 가능.

### 5.5 반복 수합

- 언제든 Phase 2·3 진행 중에도 Phase 1으로 돌아와 자료 추가 가능. 추가 시 매칭 규칙 동일 적용.

## 6. Phase 2 — 상대평가 등급 부여

- **실행 시점**: 교사가 결정(버튼). 신규 제출물·체크박스 변경 발생 시 "재계산 필요" 배지 표시.
- **파이프라인**:
  1. 제출물 단위 LLM 채점 (루브릭 기반, temperature 0, 기준별 근거 인용 필수, 원문 출력 감사용 보관)
  2. 학생별 합성 점수 (합/평균/가중 중 프로젝트 설정)
  3. 합성 점수로 전체 순위
  4. 누적 비율로 등급 파생
- **등급 매핑**:
  - 5등급 = 누적 10 / 34 / 66 / 90 / 100 (%)
  - 9등급(스테나인) = 누적 4 / 11 / 23 / 40 / 60 / 77 / 89 / 96 / 100 (%)
  - 프로젝트 설정에서 토글, 화면 즉시 반영.
- **동점자 처리**: 프로젝트 설정 (동점자 상위 등급 부여 / 중간석차 방식).
- **수동 조정**: 원 명세의 "점수를 수동으로 조정해 순위를 바꾸는" 방식은 **채택하지 않는다**. 대신:
  - 점수는 항상 제출물 채점에서 파생 재계산한다.
  - 교사 개입이 필요하면 학생별 `score_override` 필드를 둔다 (사유 필수, 감사 로그).
  - 등급을 직접 저장·수정하는 것은 금지(INV-6).

## 7. Phase 3 — 생기부 작성 (핵심 과제)

### 7.1 서술 원칙

- 산출물 내용 요약이 아니라 **학생의 경험 중심**: 동기 → 과정·노력·사고 과정 → 결과 → 배움·변화.
- 산출물과 교사 메모에 **근거가 있는 내용만** 서술.

### 7.2 생성 격리 (혼입 방지의 구조적 보장)

- **INV-1**: 생기부 생성 LLM 호출 1회에는 정확히 한 명의 학생 데이터만 포함된다. 일괄 생성도 내부적으로는 학생별 순차 호출이다.
- **INV-2**: 생성 컨텍스트는 서버가 `student_id` 필터로 DB에서 직접 조립한다. 클라이언트가 임의 텍스트를 주입할 수 없다 (교사 메모 필드는 예외이며 해당 학생 레코드에 귀속된 것만).
- **INV-3**: 생성 결과에는 근거로 사용한 제출물 id 목록(sources)을 함께 저장한다.

### 7.3 검증 패스

- 생성 직후 별도 LLM 호출(검증용 모델)로 초안을 문장 단위 분해 → 각 문장이 해당 학생의 제출물/교사 메모에 근거하는지 판정.
- 근거 없는 문장은 UI에서 하이라이트. 교사는 해당 문장 삭제/수정/재생성 선택.

### 7.4 교사 개인 관찰 메모

- 학생 정보 옆에 상시 노출되는 입력 박스. 생성 시 컨텍스트에 포함.

### 7.5 프롬프트 프로필 (생기부 메타데이터)

- "작성 참고사항" 목록 + "금지사항" 목록.
- 화면은 좌(참고)/우(금지) 2패널, 사이 분할바 드래그로 폭 조절. 항목 추가/수정/삭제 가능.
- **프로필 계층**: 계정 기본 프로필(모든 프로젝트에 적용) + 프로젝트별 오버라이드. 계정별로 따로 관리·저장.
- **예시 생기부 인제스트(프롬프트 업그레이드)**: 교사가 좋은 생기부 예시 텍스트를 입력 → LLM이 분석해 참고/금지 항목 추가·수정안을 **diff 형태로 제안** → 교사가 승인한 항목만 반영. **자동 반영 금지.**
- **문체 기본값(시드 프로필에 포함)**: 종결어미 '-함/-임/-됨', 학생 성명·인칭대명사 미표기 등.
- **버전·이력·MD 입출력 (2026-07-10 추가, 사용자 승인)**: 프로필을 저장/예시반영/가져오기/복원할 때마다 **버전 번호를 올리고 업데이트 날짜시간을 기록**해 화면에 표시한다. 현재 프로필을 버전·날짜 헤더가 붙은 **Markdown으로 내보내기**하고, 편집한 Markdown을 **미리보기 후 가져오기**할 수 있다. **과거 버전 열람·복원**을 제공한다(이력은 append-only, 복원=새 버전 저장). 근거: DECISIONS 2026-07-10.

### 7.6 글자수 카운터

- 프로젝트 설정의 제한값 대비 실시간 표시.
- 카운트 방식 토글: 글자수(공백 포함) / 바이트(한글 3바이트).

## 8. 결과 화면 (Phase 3 표)

- **표 열**: 학생 정보(이름·학번) | 등급 | 교사 관찰 메모 | 생성된 생기부.
- 열 너비 드래그 조절.
- 텍스트 셀(메모, 생기부)은 3가지 표시 모드: 접기(1줄) / 전체 맞춤(내용 전체 높이) / 커스텀(드래그로 높이 조절). 행 단위 토글 + 전체 일괄 토글 제공.
- 조정한 레이아웃은 (사용자, 프로젝트) 단위로 DB에 저장(디바운스), 재로그인 시 복원.

## 11. 확정된 주요 결정

- **인증**: Google OAuth (email·profile 기본 스코프만 → 앱 검증 불필요)
- **OCR**: LLM 비전 (등록된 API 키 재사용, purpose='추출' 모델 라우팅)
- **구글 시트**: xlsx/CSV 내보내기 업로드 (API 연동 없음). 재업로드 시 중복·변경 감지 필수
