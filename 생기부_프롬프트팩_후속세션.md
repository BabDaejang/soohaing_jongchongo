# 생기부 작성 웹앱 — 후속 세션 프롬프트 팩 (세션 4 ~ 세션 9)

버전 v2.0 · 2026-07-08 · 세션 0~2 완료 이후의 나머지 단계 · Claude Fable 5 (Claude Code / VS Code)

> 이 문서는 이미 완료된 세션 0~2 이후에 이어붙이는 후속 팩입니다. 원 팩의 **공통 프리앰블**과 **세션별 분할 구조**를 그대로 유지하되, 세션 프롬프트는 실제로 구축된 코드베이스(Next.js 16 / `proxy.ts` / `lib/llm`·`lib/crypto`·`lib/audit` / supabase CLI 마이그레이션)를 재사용하도록 다시 썼습니다.

## A. 진행 대조 — 원 팩 vs 실제 진행

세션 0이 생성한 PROGRESS.md가 원 팩과 다른 구획으로 재편되었고, 실행 중 병합·분할이 일어났습니다. 아래가 최종 정렬입니다.

| 원 팩 세션 | 실제 진행 | 상태 |
|---|---|---|
| S0 프로젝트 헌장 | 세션 0 | ✅ 완료 |
| S1 스캐폴드 + 인증 + 승인 | 세션 1 | ✅ 완료 (마이그레이션은 profiles·app_settings만, 관리자 UI는 S2로) |
| S2 관리자 패널 + API 키 + LLM | 세션 2 | ✅ 완료 (세션0 PROGRESS 기준 S2+S3 병합) |
| S3 프로젝트 CRUD + 전체 스키마 + RLS | → **세션 4** | 예정 (전체 스키마 일괄 대신 기능별 증분 마이그레이션) |
| S4 Phase 1 인제스트(일괄) | → **세션 5 + 세션 6** | 예정 (업로드·파싱·중복 / 매칭·큐·원본삭제로 분할) |
| S5 Phase 2 평가 | → **세션 7** | 예정 |
| S6 Phase 3 생기부 + 프로필 | → **세션 8a** | 예정 |
| S7 결과 테이블 UI | → **세션 8b** | 예정 |
| S8 통합 QA + 배포 | 재편된 계획에서 **누락** → **세션 9로 복원** | 예정 (아래 J절, 이유는 DECISIONS 기록 권장) |

**핵심 변화 3가지**
1. 스키마를 세션 3에서 한꺼번에 만들지 않고, 각 세션이 필요한 테이블만 마이그레이션으로 추가합니다(증분). 그래서 세션마다 "마이그레이션 + RLS + 타 계정 접근 차단 테스트"가 세트로 들어갑니다.
2. Phase 1이 업로드·파싱·중복(세션 5)과 매칭·확인 큐·원본삭제(세션 6)로 나뉩니다. 혼입 방지의 결정적 규칙은 세션 6에 집중됩니다.
3. 원 팩 세션 8(QA·배포·불변조건 감사)이 재편 계획에서 빠졌습니다. 생략하면 위험하므로 세션 9로 되살렸습니다.

**이미 구축되어 재사용할 자산 (모든 후속 세션이 새로 만들지 말고 이것들을 사용)**
- 마이그레이션 워크플로: `supabase/migrations/`에 새 파일 추가 → `supabase db push --db-url "$SUPABASE_DB_URL"` → `pg`로 검증
- RLS 헬퍼: `is_admin()`, `is_approved()` (0001에서 생성). 신규 테이블은 소유자 기반 + admin 전체 열람 패턴을 답습
- `lib/llm/`: `callLLM({ userId, purpose, messages, routing })` — purpose(추출/매칭/평가/생성/검증)로 모델 라우팅. `resolveApiKey`(개인>기본>에러). 전부 server-only
- `lib/llm/routing.ts`의 `DEFAULT_MODELS` + 시드 프로바이더 id(google/anthropic/openai, 이름으로 조회)로 프로젝트 `model_routing` 기본값 조립
- `lib/crypto/`(AES-256-GCM), `lib/audit.ts`(`writeAuditLog`, service role)
- `proxy.ts`(Next 16 미들웨어 규약): 신규 보호 라우트 가드는 여기에 추가
- 서버 전용 모듈 테스트: `node:test` + `tsx`, `--conditions=react-server` (`npm test` 형태 유지)

## B. 공통 프리앰블 (모든 후속 세션 프롬프트 맨 앞에 그대로 붙임 — 원 팩과 동일)

```text
[공통 작업 규약 — 반드시 준수]

너는 이 저장소에서 여러 세션에 걸쳐 진행되는 프로젝트의 시공자다. 이번 세션은 전체 중 일부만 담당한다.

1. 작업 시작 전 반드시 다음 파일을 읽어라: CLAUDE.md, docs/SPEC.md, docs/DATA_MODEL.md, docs/PROGRESS.md, docs/DECISIONS.md
2. docs/SPEC.md가 단일 진실 원천이다. 이번 세션 지시와 SPEC이 충돌하면 구현을 멈추고 어느 쪽을 따를지 나에게 질문하라.
3. SPEC에 명시되지 않은 설계 판단을 내렸다면, 그 즉시 docs/DECISIONS.md에 "날짜/판단/이유" 형식으로 기록하라.
4. 이번 세션의 범위 밖 코드는 수정하지 마라. 리팩터링이 필요해 보이면 제안만 하고 DECISIONS.md에 기록하라.
5. SPEC의 불변 조건(INV-1 ~ INV-6)은 어떤 경우에도 위반하지 마라. 특히 학생 데이터 혼입 가능성이 있는 로직(매칭, 생기부 생성)은 보수적으로 설계하라.
6. 모르는 것, 애매한 것은 추측하지 말고 나에게 질문하라. 특히 외부 서비스(Supabase, Vercel) 설정값이 필요하면 요청하라.
7. 세션 종료 절차: (a) typecheck·build 통과 확인 (b) 이번 세션 수용 기준 자체 점검 결과 보고 (c) docs/PROGRESS.md 체크 갱신 (d) 의미 있는 메시지로 git commit
8. API 키, 시크릿은 절대 코드에 하드코딩하지 마라. .env.local과 .gitignore를 확인하라.

먼저 플랜 모드로 이번 세션의 구현 계획을 제시하고, 내 승인을 받은 뒤 코딩을 시작하라.
```

> 후속 팩 공통 유의: 각 세션은 신규 파일을 창작하기 전에 **기존 `lib/`(llm·crypto·audit·supabase)와 `proxy.ts`, 마이그레이션 규약을 먼저 확인하고 재사용**하라. 위 A절의 "재사용할 자산" 목록을 근거로, 이미 있는 것을 다시 만들지 마라. (이 문장을 각 세션 프롬프트 하단에도 넣어 두었습니다.)

## C. 세션 4 — 프로젝트 · 학생 · 루브릭 기초

```text
[세션 4] 프로젝트/학생/루브릭 기초. 로그인 후 첫 화면부터 프로젝트 설정까지. 전체 스키마를 한 번에 만들지 말고 이번 세션에 필요한 테이블만 추가한다.

범위:
- 마이그레이션 0003_projects_students_rubrics.sql:
  - projects(owner=auth.uid, 이름, 설명, 설정 컬럼: 등급제 5/9, 글자수 제한, 카운트 방식(char/byte), 합성 방식(sum/avg/weighted), 동점자 처리, 원본 삭제 정책, model_routing jsonb)
  - students(project_id FK, student_no, name, 교사 관찰 메모 text, UNIQUE(project_id, student_no))
  - rubrics(project_id FK, criteria jsonb: [{name, description, max_score, weight}])
  - RLS: 소유자만 자기 프로젝트/하위 행 접근, admin 전체 열람. 0001의 is_admin()/is_approved() 재사용.
  - supabase db push --db-url "$SUPABASE_DB_URL" 적용 후 pg로 검증.
- 프로젝트 목록 화면(로그인 후 첫 화면): 생성/수정/삭제, 카드 클릭 → 프로젝트 홈.
- 프로젝트 홈: Phase 1/2/3 단계 내비게이션(교사가 처음 써도 흐름을 알 수 있는 안내 문구 포함).
- 프로젝트 설정 화면: 위 설정 컬럼 편집 UI. model_routing 기본값은 lib/llm/routing.ts의 DEFAULT_MODELS + 시드 프로바이더 id(이름 조회)로 조립하라(직접 새 상수 만들지 말 것).
- 루브릭 편집 UI: 기준 CRUD(이름/설명/배점/가중치). 합리적 기본 루브릭 시드(예: 과제 이해도 / 탐구 과정의 구체성 / 사고의 깊이 / 표현·완성도).
- 학생 목록: 수동 추가/수정/삭제, 학생 정보 옆 교사 관찰 메모 입력 박스(자동 저장). ※ 파일 업로드 기반 학생 생성은 세션 5~6 담당.
- proxy.ts: 프로젝트 관련 라우트는 approved 사용자만 접근하도록 가드 추가.

범위 제외(다음 세션): 파일 업로드/파싱/매칭, 제출물 테이블.

수용 기준:
1. 프로젝트 생성→수정→삭제, 설정 저장/재로드 왕복 정상 동작.
2. RLS: 타 사용자 계정으로 남의 프로젝트/학생/루브릭 접근 시 차단됨(pg 테스트 스크립트로 증명).
3. 프로젝트 삭제 시 하위(students/rubrics) 처리 방침(cascade 여부)을 DECISIONS.md에 기록하고 구현.
4. model_routing 기본값이 DEFAULT_MODELS와 시드 프로바이더에서 정확히 조립됨.
5. typecheck·lint·build·test 통과.

작업 전 A절 "재사용할 자산"(lib/*, proxy.ts, 마이그레이션 규약)을 확인하고 기존 것을 재사용하라.
```

> 실행 메모(2026-07-08, 세션 4 완료): 위 설정 화면에서 **용도별 `model_routing` 편집 UI는 세션 4에서 구현하지 않고 기본값 자동 조립(`buildDefaultModelRouting`)까지만** 완료했습니다. 편집 UI는 프로바이더·모델 선택 목록이 필요해 **세션 7로 이월**(DECISIONS 2026-07-08). 세션 4의 나머지 범위(프로젝트 CRUD·설정 6항목·루브릭·학생·RLS)는 완료되었습니다.

## D. 세션 5 — Phase 1(a) 업로드 · 파싱 · 중복 감지

```text
[세션 5] Phase 1 전반부: 파일 업로드, 텍스트 추출, 재업로드 중복·변경 감지. 학생 매칭 자체는 세션 6이 담당한다. 이번 세션은 "파싱된 제출물 후보"를 스테이징 상태로 만드는 데까지다.

범위:
- 마이그레이션 0004_submissions.sql:
  - submissions(project_id FK, student_id FK NULL 허용(매칭 전), raw_student_no text, raw_student_name text, submission_ref text(제출물ID/파일명), extracted_text text, extraction_method('text_layer'|'vision_ocr'|'sheet_row'|'manual'), content_hash text, match_status('unmatched'|'auto'|'confirmed'|'update_pending' 등), reflect boolean default true, storage_path text NULL, created_at)
  - DB에는 추출·정제 텍스트만 저장. 원본 바이너리는 저장하지 않는다.
  - RLS: 소유 프로젝트 기준. supabase db push 후 pg 검증.
- Storage 임시 버킷 생성(원본 파일 임시 보관용). 접근은 소유자로 제한.
- 업로드 UI: 다중 파일 드래그앤드롭(xlsx, csv, docx, pdf(텍스트/스캔), png/jpg) → 임시 버킷.
- 파싱 모듈 lib/parsing/:
  - parseXlsx(SheetJS), parseCsv(papaparse), parseDocx(mammoth), parsePdf(텍스트 레이어 우선), parseImage
  - parsePdf: 페이지당 추출 텍스트가 임계값 미만이면 스캔본으로 판정 → 페이지 이미지 렌더 → 비전 OCR(callLLM, purpose='추출'). extraction_method 기록.
  - parseImage: 비전 OCR(callLLM, purpose='추출').
- 스프레드시트 열 매핑 UI: 헤더 미리보기 → LLM이 학번/이름/제출물ID/내용 열 초기 추천(callLLM, purpose='추출' 또는 '매칭') → 교사 확정 → 행 단위로 제출물 후보 분해.
- 구글 스프레드시트: 별도 API 연동 없음. 교사가 xlsx/CSV로 내보내 업로드하는 경로만 지원(SPEC 5절 확정).
- 재업로드 중복·변경 감지: 같은 시트를 갱신해 반복 업로드하는 패턴 전제. 키 = (raw_student_no 또는 raw_student_name) + submission_ref. 같은 키에서 content_hash가 동일하면 스킵, 다르면 match_status='update_pending'으로 표시(자동 덮어쓰기 금지). content_hash는 정규화 텍스트 해시.

수용 기준:
1. 4종(xlsx / docx / 텍스트 PDF / 스캔 PDF) 각각 업로드 → 텍스트 추출 왕복 시연. 스캔본은 vision_ocr로 처리됨.
2. 동일 xlsx를 2회 업로드 시 중복 제출물이 0건 생성됨. 일부 행 수정 후 재업로드 시 해당 행만 update_pending으로 대기(자동 반영 아님).
3. DB에 원본 바이너리가 저장되지 않음.
4. typecheck·lint·build·test 통과.

작업 전 A절 "재사용할 자산"을 확인하라. 특히 OCR은 lib/llm의 callLLM(purpose='추출')을 쓰고, 별도 OCR 서비스를 붙이지 마라.
```

## E. 세션 6 — Phase 1(b) 매칭 · 확인 큐 · 원본 삭제

```text
[세션 6] Phase 1 후반부: 학생 매칭, 확인 대기 큐, 제출물 상세, 원본 삭제. 이 세션은 학생 데이터 혼입 방지의 결정적 방어선이다. SPEC 5절 매칭 규칙 (a)~(d)를 문구 그대로 구현하라.

범위:
- 매칭 엔진(결정적 규칙 우선, LLM 보조):
  (a) 학번(raw_student_no) 완전 일치 → 기존 학생 자동 귀속(student_id 설정, match_status='auto')
  (b) 이름만 일치(학번 미검출) → 자동 귀속 금지. 확인 대기 큐로(동명이인 보호).
  (c) 식별 정보 미검출 → LLM이 내용에서 후보 제안(callLLM, purpose='매칭')하되 반드시 교사 확인 후 귀속.
  (d) 신규 학생 생성은 학번 신규 검출 시에만 자동. 그 외는 교사 확인.
  - 어떤 경로로 귀속됐는지 submissions.match_method에 기록.
- 확인 대기 큐 UI: 후보(기존 학생 X에 귀속 / 신규 학생 생성 / 보류)를 파일·행별로 제시, LLM 후보는 근거와 함께 표시, 교사가 확정. 확정 시 match_status='confirmed'.
- 제출물 상세 화면: 학생별 제출물 목록·내용(추출 텍스트) 열람, 제출물마다 반영 체크박스(기본 on), 수정/삭제/수동 추가(텍스트 직접 입력, extraction_method='manual').
- 원본 삭제 흐름(INV-5): 제출물 상세에서 교사가 "추출 확인"을 누른 뒤에만 Storage 원본 삭제. 확인 전 삭제 금지. 프로젝트 설정의 "N일 후 자동 삭제"도 보조 정책으로 구현.
- Phase 2·3 진행 중에도 이 화면으로 돌아와 자료 추가 가능(매칭 규칙 동일 적용).

수용 기준:
```

- **세션 8a(생성)**: 일괄 생성이 내부적으로 학생별 단일 호출인지 코드 구조로 확인. 배열 시그니처 생성 함수가 생기면 INV-1 위반.
- **증분 스키마**: 각 세션이 테이블을 나눠 만들므로, 세션마다 RLS와 타 계정 차단 테스트를 빠뜨리지 않아야 함. 세션 9의 전수 점검은 이를 마지막으로 잡는 그물.
